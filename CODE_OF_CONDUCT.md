# Code of Conduct

## Our Commitment

We are committed to providing a welcoming, professional, and secure environment for all contributors, maintainers, and users of this NIST SP 800-218 compliant project.

## Our Standards

### Professional Conduct

Examples of behavior that contributes to a positive environment:

- **Respectful communication** in all interactions
- **Constructive feedback** on code and security practices
- **Collaborative problem-solving** for security challenges
- **Professional handling** of disagreements
- **Inclusive language** that welcomes all contributors
- **Focus on project goals** and security objectives

### Security-Specific Standards

Given this project's security focus, additional standards apply:

**Responsible Security Behavior**

- **Confidential handling** of security vulnerabilities until patches are available
- **Professional discussion** of security topics without creating fear or panic
- **Accurate representation** of security risks and mitigations
- **Respect for security policies** and procedures
- **Timely response** to security-related communications
- **No public disclosure** of vulnerabilities before coordination with maintainers

**Security Knowledge Sharing**

- **Share security knowledge** constructively to improve overall project security
- **Mentor newcomers** in secure development practices
- **Provide context** when suggesting security improvements
- **Document security decisions** clearly for future contributors

### Unacceptable Behavior

Examples of unacceptable behavior include:

- Harassment, discrimination, or intimidation of any form
- **Public disclosure** of security vulnerabilities before proper coordination
- **Intentional introduction** of security vulnerabilities or malicious code
- **Sharing of credentials** or sensitive security information inappropriately
- **Disrupting security processes** or ignoring security requirements
- Personal attacks or unprofessional language
- Spam, trolling, or other disruptive behavior

## Security Incident Reporting

### Security Vulnerabilities

**Never report security vulnerabilities through public channels.**

- **Report to**: <security@scanset.io>
- **Process**: Follow responsible disclosure in SECURITY.md
- **Response time**: We respond within 48 hours
- **Coordination**: We will work with you on disclosure timeline

### Code of Conduct Violations

Report violations of this code of conduct to:

- **Primary contact**: <conduct@scanset.io>
- **Secondary contact**: <security-team@scanset.io> (for security-related violations)

## Our Responsibilities

### Project Maintainers

Project maintainers are responsible for:

- **Clarifying standards** of acceptable behavior
- **Enforcing consequences** for unacceptable behavior
- **Maintaining confidentiality** of security reports
- **Coordinating security** disclosure processes
- **Providing guidance** on security best practices
- **Creating inclusive environment** for security learning

### Security Team

The security team has additional responsibilities:

- **Triaging security reports** professionally and promptly
- **Maintaining confidentiality** during vulnerability assessment
- **Coordinating with reporters** on responsible disclosure
- **Providing security guidance** to contributors
- **Ensuring compliance** with security standards

## Enforcement

### Process

1. **Initial Assessment**: Conduct violations will be reviewed by maintainers
2. **Investigation**: Fair and thorough investigation of reported incidents
3. **Response**: Appropriate action based on the nature and severity of violation
4. **Communication**: Clear communication of decisions and rationale
5. **Appeal Process**: Opportunity for appeal of enforcement decisions

### Consequences

Consequences for unacceptable behavior may include:

**Warning**: Private written warning with clarification of violation
**Temporary Suspension**: Temporary ban from project participation
**Permanent Ban**: Permanent exclusion from project participation
**Security Access Revocation**: Removal of access to security-sensitive information
**Legal Action**: For malicious security violations or legal violations

### Security-Specific Consequences

Additional consequences for security-related violations:

**Security Access Removal**: Immediate revocation of access to security processes
**Vulnerability Report Ban**: Prohibition from reporting security vulnerabilities
**Security Team Exclusion**: Removal from security-related discussions
**Public Notification**: In cases of malicious security behavior (after legal review)

## Scope

This Code of Conduct applies to:

- **All project spaces**: GitHub repositories, discussions, issues, pull requests
- **Communication channels**: Email, chat, forums related to the project
- **Public representation**: Conferences, meetups, social media when representing the project
- **Security processes**: Vulnerability reporting, security reviews, incident response

## Attribution and Enforcement Authority

This Code of Conduct is enforced by the project maintainers and security team.

**Primary Enforcement**: Project Maintainers
**Security Enforcement**: Security Team
**Escalation**: [Organization] Security Office
**Legal Issues**: [Organization] Legal Counsel

## Reporting Guidelines

### When to Report

- Harassment, discrimination, or unprofessional behavior
- Security policy violations or misuse of security processes
- Violations of confidentiality regarding security matters
- Any behavior that creates an unsafe or unwelcoming environment

### How to Report

1. **Email**: <conduct@scanset.io>
2. **Include**: Detailed description, evidence, and desired resolution
3. **Confidentiality**: All reports are handled confidentially
4. **Response**: You will receive acknowledgment within 24 hours

### What Happens Next

1. **Acknowledgment**: We confirm receipt of your report
2. **Investigation**: Fair and thorough review of the incident
3. **Decision**: Appropriate action based on findings
4. **Follow-up**: Communication of resolution and any changes needed

## Security Culture

We strive to maintain a security culture that:

- **Values security** as a fundamental requirement, not an afterthought
- **Encourages learning** about secure development practices
- **Supports responsible** vulnerability disclosure
- **Promotes collaboration** on security improvements
- **Maintains professionalism** in security discussions
- **Protects confidentiality** of security-sensitive information
